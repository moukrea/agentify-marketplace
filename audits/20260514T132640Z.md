## Findings

### F-001 — major / manifest-drift: bats manifest-conformance over-restricts plugin.json after schema-alignment fix

`tests/manifest-conformance.bats` asserts that `plugin.json` must declare a non-empty `commands[]` array (test 5, L36-38), a `hooks` field (test 6, L41-46), and a command entry per skill directory (test 7, L49-59). The Claude Code plugin manifest schema (verified against `https://www.schemastore.org/claude-code-plugin-manifest.json` — only `name` is required; `commands` and `hooks` are explicitly optional; `skills` may be a string path) makes none of these mandatory. After PR #7 (commit `b50faeb`) correctly slimmed `plugin.json` to schema-conformant shape (declaring only `skills: "./skills"`), those three bats assertions block on every push. The schema is right; the tests are wrong.

- 3/20 assertions fail locally (`bats tests/manifest-conformance.bats`):
  - `not ok 5 plugin manifest declares a commands array`
  - `not ok 6 plugin manifest declares the hooks manifest path`
  - `not ok 7 every skill directory has a matching command entry` (16 missing entries fabricated by the loop)
- This is the immediate cause of the `manifest-conformance` CI job being red on every push since 4f3c096; the broader CI breakage is independent and tracked under F-002.

### F-002 — critical / ci-broken: three workflow files have YAML literal-scalar breaks at zero-indent heredoc/python content

`gh run list --workflow ci.yml --branch main --limit 10` returns **only failures** — every ci.yml push run on `main` is red. The dispatcher reports `27/28 = 3.57 %` success across the last 28 main runs vs. the 90 % skill threshold. The three workflow files that fail at workflow-validation stage (0 jobs scheduled, GitHub Actions UI cannot render the workflow graph, the run displays the path as the workflow name instead of the declared `name:` value) all share the same root cause:

- `.github/workflows/ci.yml` L91-98: inline `python3 -c "...` heredoc body indented at column 1 inside a `run: |` literal block scalar whose first content line (L25 `set -euxo pipefail`) lives at column 11. Per the YAML 1.2 literal-style spec (verified at `https://yaml.org/spec/1.2.2/#literal-style`), a content line with less indentation than the first content line terminates the scalar — so PyYAML raises `ScannerError: could not find expected ':'` at line 92.
- `.github/workflows/audit-trend.yml` L55-63: `gh pr create --body "$(cat <<'EOF' ... EOF)"` heredoc body at column 1 — same failure pattern (PyYAML breaks at L55, col 1).
- `.github/workflows/practice-evolve.yml` L65-67: same heredoc-at-column-1 pattern (PyYAML breaks at L65, col 1).

`changelog-pr.yml`, `auto-approve-bot-prs.yml`, `release.yml`, `feedback-triage.yml`, `self-improve.yml` either do not contain a zero-indent heredoc body OR were never modified to introduce one (those workflows show their declared `name:` correctly in the workflows API listing, and their runs schedule jobs). Confirming evidence: the workflows API returns `"name":".github/workflows/ci.yml"` for the three broken ones and `"name":"changelog-pr"` etc. for the working ones — GitHub falls back to the file path when it cannot parse the workflow's `name:` key.

Fix shape: indent every content line of the heredoc/python body to at least column 11 (matching the `run: |` scalar's expected indentation). Alternative: switch to an external script file invoked from the `run:` step.

### F-003 — moderate / bug: feedback_ingest.sh uses invalid `?.` jq syntax (silently kills every feedback ingest)

`plugins/agentify/lib/feedback_ingest.sh:167` has `| capture("agentify-feedback-id: (?<id>[a-f0-9-]+)")?.id`. jq does not support the `?.` optional-chaining operator (JavaScript syntax, not jq). Running `bash plugins/agentify/lib/feedback_ingest.sh` against the live upstream repo dies with `jq: error: syntax error, unexpected FIELD (Unix shell quoting issues?) at <top-level>, line 5: | capture("agentify-feedback-id: (?<id>[a-f0-9-]+)")?.id`. That means every downstream consumer (`/agt-self-improve`'s feedback phase, `/mkt-self-improve` Phase 5, `feedback-triage.yml`) silently degrades — and Phase 5 of this very audit had to bypass `feedback_ingest.sh` and query `gh` directly.

Fix shape: replace with `| (capture("agentify-feedback-id: (?<id>[a-f0-9-]+)") // {}).id // null`.

### F-004 — moderate / feedback-recurring: issue #1 has aged 16 days without triage

`https://github.com/moukrea/agentify-marketplace/issues/1` ("renderer: bin/agentify substitution find-scope corrupts plugin source on self-host", opened 2026-04-28) has no labels (no `triage`, no `addressed`, no `wontfix`) and the body documents a **data-loss bug**: `bin/agentify` runs its placeholder-substitution `find` over the entire `--output` tree, so when output==marketplace-root (the self-host case) it rewrites the authored plugin templates in place and destroys the source-of-truth. The reporter listed 7 specific files corrupted on the last self-host attempt and proposed a patch that scopes the `find` to recently-copied paths. Per the skill spec, any open feedback issue ≥7 d without `triage`/`addressed`/`wontfix` is a feedback-recurring finding.

### F-005 — info / meta: practice-evolve infrastructure is wired but never bootstrapped

`plugins/agentify/conventions/pinned-practices.json` has `"sources": {}` (empty) — no source has ever been seeded with recommendations, so `/mkt-practice-evolve`'s adoption-check phase has nothing to assert against. The 13 sources declared in `sources.yaml` are fetchable (`practice_track fetch agents-md-spec` and `practice_track fetch anthropic-claude-code-docs` both succeeded in this run, writing `plugins/agentify/practices/raw/<id>/2026-05-14.md`), but the `practice-evolve.yml` workflow has been failing at workflow-validation stage on every push (covered by F-002), so the weekly cron has never produced a baseline. Until pinned-practices.json carries at least one recommendation per high-authority source (Anthropic, Shopify, Karpathy at minimum), Phase 8 of this skill is structurally a no-op.
