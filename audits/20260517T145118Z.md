### F-002 — moderate / meta: `mkt-self-improve` Phase 3's 90 % rule treats `cancelled` as non-success; today's CI is 100 % failure-free yet still scores 83 %

`bash plugins/agentify/lib/git_host.sh ci_status main 30` returns 30
runs with conclusions `25 success + 5 cancelled + 0 failure`. Every
cancellation is the correct behaviour of
`concurrency.cancel-in-progress: true` (see
`.github/workflows/ci.yml`'s `concurrency:` block, and the matching
block in `changelog-pr.yml` which the prior audit's F-001
investigated) — when two pushes land on `main` within seconds of each
other, the older in-flight run is cancelled before its job loop
starts. Cancellations of this shape are not failures; they are CI
correctly recognising that the newer commit supersedes the older
one. The skill's literal rule (`<90 % success → severity: major
finding under category: ci-broken`) does not distinguish, so a
healthy CI scores `degraded`. This is a meta-finding about the
audit-skill rule itself, not about CI: the skill's own
`mkt_self_improve_postflight.sh` already enforces nuanced gates
(20 %-sample re-fetch, hostname diversity); Phase 3's metric should
match that sophistication.

Fix shape: either (a) the skill spec at
`.claude/skills/mkt-self-improve/SKILL.md` updates the Phase 3 rule
to `<90 % of (success + failure) runs are success` (denominator
excludes `cancelled`, `neutral`, `skipped`), OR (b) `git_host.sh
ci_status` gains a `--exclude-cancelled` flag and `/mkt-self-improve`
calls it that way. Approach (a) is one-line in the skill spec;
approach (b) is one-flag in the lib script. Both are mechanical.

- Acceptance criterion: the skill spec updates the Phase 3 rule (or
  `git_host.sh ci_status` gains an exclude-cancelled mode that
  `/mkt-self-improve` invokes) AND a subsequent
  `/mkt-self-improve` reports Phase 3 as `pass` when there are zero

### F-005 — major / practice-drift: Agent Teams is a structural peer to `/agt-loop`'s REVISE+REVIEW subagent shape; the harness uses `subagent_type: general-purpose` per LOOP_PROMPT.md and ignores agent-team primitives

`plugins/agentify/LOOP_PROMPT.md:12` declares:

```
- Spawn subagents via the Agent tool (`subagent_type: general-purpose`,
  fresh context). Pass each a fully-formed prompt as written below.
  Capture the reply.
```

— the REVISE and REVIEW subagents that drive every loop iteration
are spawned as anonymous `general-purpose` subagents whose results
return through the parent's `Agent` tool call. This is exactly the
shape the `agent-teams.md` upstream doc describes as **subagent-based
parallelism** and contrasts with **agent-team** parallelism:

> "Subagents only report results back to the main agent and never
> talk to each other. In agent teams, teammates share a task list,
> claim work, and communicate directly with each other."

The harness's REVISE→REVIEW chain is sequential by design (REVIEW
depends on REVISE's output, per `LOOP_PROMPT.md:24`), so peer
messaging is not a hard requirement today. But three agent-team
features are structurally useful for the loop and inaccessible from
inside a `general-purpose` subagent:

1. **`TaskCreated` / `TaskCompleted` / `TeammateIdle` hooks**
   (already enumerated in `claude-code-mechanics.md:20` and in the
   hook event taxonomy). Today the loop's
   `${state_root}/loop-state.json` is the orchestrator's own state
   file; with an agent team, the convergence-detection logic in
   `LOOP_PROMPT.md §C7` could fire a `TaskCompleted` hook
   declaratively rather than the orchestrator polling.
2. **Subagent definitions** (`agent-teams.md`'s "Use subagent
   definitions for teammates" section). The agentify plugin ships
   `plugins/agentify/agents/` but only as documentation; teammates
   could `Spawn a teammate using the revise agent type` and inherit
   the agent definition's `tools` allowlist + model selection,
   replacing the inline-prompt-construction in `LOOP_PROMPT.md §C2 /
   §C3`.
3. **Plan-approval gating** (`agent-teams.md`'s "Require plan
   approval for teammates"). The harness's v6.0 plan-mode
   integration (per memory: `project_plan-mode-integration`) is
   wired into `/agt-plan` for the parent; a REVISE teammate could
   work in read-only plan mode and submit its plan to the REVIEW
   teammate or back to the orchestrator for approval.

Acting on this finding requires an ADR, not a refactor. Agent teams
ship with hard constraints: they require
`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`, Claude Code v2.1.32+, and
have documented known limitations including "No session resumption
with in-process teammates" and "Task status can lag." For a harness
plugin whose value proposition is reliability, building on
experimental machinery is a deliberate decision that needs ADR
discipline. Anthropic's own engineering experiment running 16
parallel Claude Opus 4.6 agents to build a Rust-based C compiler
demonstrates the scale agent teams target (2,000 sessions over two
weeks, ~$20,000 in API cost), which is the production-grade end of
the spectrum.

- Acceptance criterion: `decisions/` carries either an accepted ADR
  recording a deliberate migration plan from
  `subagent_type: general-purpose` REVISE+REVIEW to an agent-team
  shape (with concrete `~/.claude/teams/agentify-loop/config.json`
  template + a flag in `agentify.config.json` to opt in), OR an
  accepted ADR recording the deliberate decision NOT to migrate
  citing the agent-teams known-limitations as the blocker AND
  `claude-code-mechanics.md` gains a "Why we use subagents not
  agent teams" subsection.
