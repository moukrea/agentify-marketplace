## Findings

### F-001 — major / ci-broken: `changelog-pr.yml` fails 100 % of push runs (bot branch already exists, `git push` non-fast-forwards)

The most-recent three `changelog-pr` runs on `main` (HEAD `707fadd`,
`a87de32`, `4f7a0df`) all fail at the same step ("Open PR if CHANGELOG
diff is non-empty"). The failure tail (run 25882708504, log line at
20:05:33.4148) is `! [rejected]        bot/changelog-2026-05-14 -> bot/changelog-2026-05-14 (non-fast-forward)`. Root cause: the workflow
computes the branch name as `bot/changelog-$(date -u +%Y-%m-%d)`
deterministically, so two same-day pushes to `main` both try to
`git push -u origin "$branch"` against a branch that already carries
the first push's commit. `concurrency.cancel-in-progress: true`
(`changelog-pr.yml:14-16`) prevents two runs from being _in flight_
simultaneously but does nothing about the second run's push collision
once the first run has already pushed and exited. The result: CI
shows green for `ci.yml` but the `main` branch's run-list reads 23
failures of 30 (one cancelled).

Fix shape (preferred): on push collision, `git push --force-with-lease`
to the dated branch (safe because the bot owns it exclusively), OR
append `-attempt-$(uuidgen | head -c 8)` to disambiguate, OR fetch +
rebase before pushing.

- Acceptance criterion: 10 consecutive `changelog-pr.yml` runs on
  `main` conclude `success` (or `skipped` for "no changelog diff;
  nothing to PR"), measured via
  `bash plugins/agentify/lib/git_host.sh ci_status main 30`.

### F-004 — moderate / practice-drift: `plugins/agentify/.claude-plugin/plugin.json` declares no `dependencies[]` despite Claude Code v2.1.110+ supporting tag-resolved semver dependencies

`plugins/agentify/.claude-plugin/plugin.json` (32 lines) declares
`name`, `version: "6.0.0"`, `skills: "./skills"`, and metadata fields
— but no `dependencies` array. Claude Code v2.1.110 added first-class
plugin-dependency resolution: `dependencies` accepts either a bare
string or an object with `{name, version, marketplace}`, where
`version` is a Node semver range (`~2.1.0`, `^2.0`, etc.) resolved
against tags shaped `{plugin-name}--v{version}` on the upstream
repository. The companion CLI helper `claude plugin tag --push`
generates the tag deterministically from `plugin.json`. The same docs
introduce `allowCrossMarketplaceDependenciesOn` for the root
`marketplace.json`, the `dependency-unsatisfied` /
`dependency-version-unsatisfied` / `no-matching-tag` error codes, and
`claude plugin prune` (v2.1.121+) for orphaned-dependency cleanup.

The marketplace's `bin/bump-version.sh` syncs
`plugin.json:.version` + `marketplace.json` and creates an annotated
tag, but the tag is shaped `vX.Y.Z` (release-style), not
`agentify--vX.Y.Z` (plugin-dependency-resolution-style). Today the
plugin has no downstream dependents, so no consumer breaks; the gap
is forward-looking: the moment a fleet plugin declares `"agentify"`
in its `dependencies[]`, version resolution silently falls back to
"latest available" because no `agentify--v*` tag exists.

- Acceptance criterion: `bin/bump-version.sh` produces both the
  release-style `vX.Y.Z` annotated tag AND the plugin-resolution-style
  `agentify--vX.Y.Z` tag in a single `--push` (or documents the
  separation explicitly), AND `plugin.json` carries an empty-but-present
  `dependencies: []` array as schema future-proofing.

### F-005 — moderate / practice-drift: `/agt-loop` machinery duplicates the bundled `/loop` skill + `CronCreate` / `CronList` / `CronDelete` tools shipped in Claude Code v2.1.72+

`plugins/agentify/LOOP_PROMPT.md` (27,580 bytes) plus the
`.agents-work/` path-root convention plus `bash` hooks under
`plugins/agentify/hooks/` implement the revise → review loop the
agentify harness centres on. Claude Code v2.1.72 shipped a bundled
`/loop` skill that runs a prompt on a cron-shaped interval, with the
trio `CronCreate` (5-field cron expression + prompt + recur flag),
`CronList`, and `CronDelete` exposed as tools the model can call
directly. The bundled skill also supports a dynamic mode where Claude
chooses the interval per iteration based on observed activity, and
respects a `loop.md` file at `.claude/loop.md` or `~/.claude/loop.md`
that replaces the default maintenance prompt. Recurring tasks expire
after 7 days; session-scoped scheduling pairs with `Channels` for
event-driven push (an opposite shape from interval polling).

The agentify harness has independently re-invented the interval-loop
construct under its own naming. There is no direct conflict — the
two can coexist — but `/agt-loop` cannot benefit from the upstream's
dynamic interval, channel push-events, or 7-day-expiry-by-default
hygiene without a deliberate integration. Worse, a user invoking both
in one session will see two scheduled-task lists (the agentify
`.agents-work/` log and the upstream `CronList` output), with no
single source of truth.

- Acceptance criterion: either
  (a) `plugins/agentify/skills/agt-loop/SKILL.md` documents how to
  migrate the loop body to a `.claude/loop.md` + `/loop` invocation
  (preserving the agentify revise/review prompt), OR
  (b) an ADR under `decisions/` records the deliberate decision to
  keep `/agt-loop` independent and lists the upstream features it
  forfeits in exchange.

### F-006 — moderate / practice-drift: Claude Code's `/goal` (condition-based loop) is unreferenced in lifecycle skills despite being structurally distinct from `/agt-loop`'s interval polling

`/goal` is a session-scoped wrapper around a prompt-based Stop hook:
after each turn a small fast model (defaults to Haiku) evaluates the
condition and returns yes-or-no; on "no" the loop continues. The
construct is structurally distinct from `/agt-loop`'s interval-based
revise/review: `/goal` blocks per-turn until a falsifiable condition
holds (compile, test pass, file under size budget, empty queue);
`/agt-loop` re-fires on a clock. The agentify lifecycle skills
(`agt-prd`, `agt-plan`, `agt-tasks`, `agt-implement`) all express
work as a sequence of task-level Validation criteria, which is
exactly the shape `/goal` was designed for ("`npm test` exits 0", "all
acceptance criteria hold"). The /agt-implement skill could delegate
each task's Validation gate to `/goal` and exit when the condition
holds, rather than relying on an interval loop or manual operator
attention.

The user memory at
`/home/eco/.claude/projects/-home-eco-Code-Personal-agentify-marketplace/memory/feedback_lifecycle-interactive.md`
already captures the "lifecycle skills are interactive, not
one-shot" rule. Wiring `/goal` into per-task Validation evaluation
preserves that interactivity (operator can still `/goal clear`) while
removing the manual "is it done yet" check.

- Acceptance criterion: either an ADR documents the decision to NOT
  adopt `/goal` (with motivation), OR `/agt-implement`'s SKILL.md
  describes how per-task Validation criteria are passed to `/goal`
  as the completion condition.

## Trend findings

Substantive online-research distillation of what the harness should
adopt, partially adopt, or explicitly decline. Each bullet carries an
adoption marker per PRD 0003 FR-2.

- **not adopted** — Plugin dependencies + tag-resolved semver
  resolution (`dependencies[]` in `plugin.json`,
  `{plugin-name}--vX.Y.Z` tag convention, `claude plugin tag --push`,
  `allowCrossMarketplaceDependenciesOn`, error codes
  `dependency-unsatisfied` / `range-conflict` / `no-matching-tag`,
  `claude plugin prune`) — all shipped in v2.1.110+ and degrade
  gracefully when absent. See
  [code.claude.com/docs/en/plugin-dependencies](https://code.claude.com/docs/en/plugin-dependencies).
  The marketplace's `plugin.json` declares neither the array nor the
  tag-style; see F-004 for the acceptance criterion.

- **not adopted** — `/goal` command (session-scoped Stop-hook wrapper
  that evaluates a condition after every turn against the small fast
  model). Structurally distinct from `/agt-loop`'s interval polling.
  Directly applicable to per-task Validation criteria in
  `/agt-implement`. See
  [code.claude.com/docs/en/goal](https://code.claude.com/docs/en/goal).
  Decision pending; see F-006.

- **partial** — `--worktree` flag, `EnterWorktree` / `ExitWorktree`
  tools, and `.worktreeinclude` env-file copy convention. The harness
  already exploits Agent isolation worktrees (subagent worktrees
  auto-cleanup at session start once orphaned, gated by
  `cleanupPeriodDays`), but the `.worktreeinclude` file (`.gitignore`
  syntax, copies tracked-AND-gitignored files into each worktree) and
  the `worktree.baseRef` setting (`fresh` vs `head` — controls
  whether new worktrees branch from `origin/HEAD` or local HEAD) are
  unreferenced. See
  [code.claude.com/docs/en/worktrees](https://code.claude.com/docs/en/worktrees).

- **not adopted** — Sandboxing (`/sandbox` command, OS-level
  Seatbelt/bubblewrap enforcement, `sandbox.filesystem.allowWrite` /
  `denyRead` / `allowRead`, `sandbox.network.allowedDomains`,
  `sandbox.failIfUnavailable`, the `dangerouslyDisableSandbox`
  escape-hatch and its `allowUnsandboxedCommands: false` lock-down).
  The agentify harness runs autonomous shell commands as a core
  loop construct, yet ships no recommended sandbox policy template
  for downstream tenants. See
  [code.claude.com/docs/en/sandboxing](https://code.claude.com/docs/en/sandboxing).

- **partial** — `/loop` bundled skill + `CronCreate` / `CronList` /
  `CronDelete` tools (v2.1.72+). The agentify `/agt-loop` was
  designed before this surface existed; the two now overlap. The
  bundled `/loop` supports a `loop.md` body file at
  `.claude/loop.md` (project) or `~/.claude/loop.md` (user) plus
  dynamic intervals; the agentify revise/review prompt could move
  into a project `loop.md` and inherit the upstream's expiry +
  channel-push hygiene. See
  [code.claude.com/docs/en/scheduled-tasks](https://code.claude.com/docs/en/scheduled-tasks).
  Decision pending; see F-005.

- **adopted** — Fast mode (`/fast`, Opus 4.6/4.7,
  `fastMode: true` user-settings field). Active in the dispatching
  session as of 2026-05-14 per the parent `/effort max` invocation.
  Today (2026-05-14) Opus 4.7 becomes the default fast mode model per
  the upstream announcement. The harness imposes no
  `fastModePerSessionOptIn` policy — fine for a single-engineer
  developer rig but worth surfacing as a managed-settings template
  for fleet tenants. See
  [code.claude.com/docs/en/fast-mode](https://code.claude.com/docs/en/fast-mode).

- **not adopted** — Routines + `/schedule` (cloud-side scheduling
  with API+GitHub-event triggers, `experimental-cc-routine-2026-04-01`
  beta header, `/fire` endpoint, per-account daily run cap). Adjacent
  to but structurally different from `/agt-loop` (cloud vs local;
  event vs interval). The fleet-bootstrap could ship a routine
  template that runs `/<prefix>-self-improve` on a weekly schedule
  on Anthropic-managed infrastructure rather than depending on a
  GitHub Actions cron that this very audit has repeatedly broken.
  See [code.claude.com/docs/en/routines](https://code.claude.com/docs/en/routines).

- **partial** — Output styles (Default / Proactive / Explanatory /
  Learning + custom styles with `keep-coding-instructions`
  frontmatter). Plugins can ship an `output-styles/` directory and
  set `force-for-plugin: true` to apply the style automatically.
  The agentify plugin ships none; lifecycle skills like
  `/agt-clarify` are inherently `Explanatory`-shaped and could
  reference the built-in style as a recommended pairing. See
  [code.claude.com/docs/en/output-styles](https://code.claude.com/docs/en/output-styles).

- **not adopted** — Agent Skills open standard
  ([agentskills.io](https://agentskills.io)). Claude Code's skills
  doc explicitly states "Claude Code skills follow the Agent Skills
  open standard, which works across multiple AI tools." 34+ adopting
  agent products are listed on the standard's homepage (Junie,
  Gemini CLI, OpenCode, Cursor, Amp, Goose, GitHub Copilot, VS Code,
  OpenAI Codex, Kiro, etc.). The agentify plugin's SKILL.md files
  mostly conform to the standard but the marketplace
  `.claude-plugin/marketplace.json` does not advertise cross-tool
  portability, and the standard's `keep-coding-instructions` /
  `paths` / `shell` frontmatter fields are unused.

- **not adopted** — v2.1.128-136 (week 19, 2026-05-04 → 2026-05-08)
  ships hook input field `effort.level` + the `$CLAUDE_EFFORT` env var
  for Bash subprocesses, the `worktree.baseRef` setting, the
  `--plugin-url` flag for ad-hoc URL-loaded plugin archives, and the
  `settings.autoMode.hard_deny` rule shape. The agentify hooks under
  `plugins/agentify/hooks/` do not branch on `$CLAUDE_EFFORT`, so a
  `max`-effort session and a `low`-effort session pass through the
  same gate thresholds. See
  [code.claude.com/docs/en/whats-new/2026-w19](https://code.claude.com/docs/en/whats-new/2026-w19).
