# Brainstorm: skill-enforcement-gates

## Problem statement

Two failure modes in the 2026-05-14 session share one root cause: SKILL.md prose describes substantive behavior, but the runtime doesn't enforce it. (1) `/mkt-self-improve` declared Phase 4 + Phase 8 complete after 1 WebFetch and 2 practice_track fetches; prose said "WebFetch all URLs cited in context/*.md… distil into distillations/… run adoption_check_command", but no structural check stopped the model from skimming. (2) `/agt-prd → /agt-plan → /agt-tasks → /agt-implement` ran back-to-back with zero user interaction despite each skill documenting pause points; 6 commits landed direct on main despite project history showing PR-based workflow. We need structural enforcement (not just stronger prose) at three classes of gate: (a) **substantive-work gate** — audit cannot finalize until N substantive tool calls happened AND a Trend-findings section is present; (b) **interaction gate** — lifecycle skills cannot persist a phase artifact until the user explicitly confirms; (c) **branch-not-main gate** — git push to main from the agent context is hard-refused.

## Open questions

- [ ] How strict the "Trend findings" content gate? Literal `## Trend findings` heading required, or per-finding `references[]` count ≥ N drawn from the authority list?
- [ ] Substantive-fetch minimum: Phase 4 ≥ 10 WebFetches against context/*.md URLs; Phase 8 ≥ 1 fetch per `authority_weight ≥ 4` source. Are these the right numbers?
- [ ] Tool-call counting mechanism: (a) state file the model maintains; (b) parse `$CLAUDE_TRANSCRIPT_PATH` if exposed to hooks; (c) the audit's `references[]` is the only source of truth (verifiable since URLs can be re-fetched at postflight time). Option (c) is cleanest but doesn't catch fetches that didn't make it into a finding.
- [ ] Branch-not-main exceptions: bot pushes to `bot/<workflow>-<date>` are already exempt (target ≠ main). Maintainer's `moukrea/<scope>/<date>-<slug>` branches also exempt. Anything else needs an allowance?
- [ ] Domain filter for "substantive" — count only fetches against `sources.yaml` + `code.claude.com` + `anthropic.com` + `github.com/anthropics/*`? Strict filter catches hype-train sources but adds maintenance burden when new authorities emerge.
- [ ] Interaction gate enforcement: does `/agt-prd` literally call `AskUserQuestion` before `task_backend prd_create`, or is "print draft + wait for freeform user reply" acceptable? The former is structurally enforceable (hook can detect the tool call); the latter is more flexible but harder to gate.

## Tentative direction

Adopt alternative 5 (**hybrid wrappers + hooks + bats**). Two artefact classes land together:

(a) `plugins/agentify/lib/agt_<skill>_preflight.sh` + `agt_<skill>_postflight.sh` — wrapper scripts SKILL.md MUST invoke. Preflights refuse when (i) HEAD is `main` AND the skill is `/agt-implement`, (ii) prior phase artifact missing, (iii) `clarifications.md` unresolved. Postflights inspect the just-written artifact for required sections + minimum citation count (Phase 4 ≥ 10 WebFetches against context-cited URLs; Phase 8 ≥ 1 fetch per `authority_weight ≥ 4` source) AND a `## Trend findings` section naming ≥3 specific upstream patterns with their adoption status in the harness. Verification: postflight re-fetches a sample of the audit's `references[]` URLs to confirm they exist and roughly match the cited content.

(b) `plugins/agentify/hooks/hooks.json` gains a `PreToolUse` hook on `Bash` (broad matcher, per #36389) running `lib/block-direct-push-to-main.sh` — parses the command, refuses with `permissionDecision: deny` when target is `origin/main` AND HEAD's history shows direct local commits on main with no maintainer-feature-branch ancestor.

Bats tests assert both layers are wired. The substantive-fetch-count rule specifically enforces the trend-discovery scope: fetches count toward the gate only when made against the curated authority list (`sources.yaml` + `code.claude.com` + `anthropic.com`), so "random hype-train blog post" fetches don't satisfy the gate.
